📦 Logistics Status Rule Engine

Shipment status transition validation · Embedded in-process library · Zero network calls

spec v1.0.0

What does this system do?
Every shipment at Logistics passes through a sequence of statuses — from DRAFT at booking, through HUB-IN-SCANNED and HUB-OUTSCAN in the middle mile, to DELIVERED or RTO_DELIVERED at the end. This library acts as the single enforcement point for every status change — if a system tries to move a shipment from an invalid state, the engine blocks it in under 1 millisecond, with no database call and no network hop. It is embedded inside every source system (Booking, Hub Ops, Delivery, 3PL) so rules are enforced even if the central platform is unavailable.

⚡ FORCE_* Overrides
What is it?
Sometimes a Hub operator needs to manually push a shipment past a stuck status — for example, forcing an outscan when the system state is inconsistent. FORCE_* statuses exist for exactly this.
How is it controlled?
Every FORCE_* transition requires an Operator ID. The engine rejects it without one. It is logged to the audit trail automatically. This keeps overrides accountable, not invisible.
What it is NOT
FORCE_* is not a bypass of the rule engine. It is a pre-approved, explicitly modelled edge in the spec — validated like every other transition. You cannot FORCE a transition the spec doesn't define.
⚠️ Exception Wildcards
What are they?
Seven exception statuses — DAMAGED, MIS_ROUTED, REROUTED, EXCESS_INSCAN, PINCODE_AUDITED, WEIGHT_AUDITED, MODE_AUDITED — can be set from any non-terminal status, in any flow. They model real operational disruptions that can happen at any point in the journey.
Why wildcards, not explicit edges?
Listing these 7 statuses as valid destinations from all ~48 non-terminal statuses would produce 336 explicit edges. Instead, they're declared once as wildcards and expanded automatically when the adjacency map is compiled at startup.

⚙️ Engine Internals — Validate() Hot Path
flowchart LR
    IN["Validate(\n  current,\n  next,\n  flow,\n  ctx\n)"]

    Z["Zero-value guard\ncurrent==0 or next==0\nErrZeroValueStatus"]

    P["Edge present?\nflagPresent bit set?"]
    O["Operator ID check\nflagRequiresOpID?\nErrMissingOperatorID"]

    IN --> Z --> U --> F --> T --> L --> P
    P -->|"miss (0)"| ERR["✗ ErrInvalidTransition"]
    P -->|"hit"| O --> IS --> R

Every guard is a single integer comparison. The adjacency map lookup is one array read. Total: ~13ns on M3 / ~21ns on server hardware.

🗃 Adjacency Map — Key Packing

All 55 statuses have stable integer IDs (1–55). The triplet (from, to, flow) is packed into a single uint16 array index. 

// Pack key: from_id << 7 | to_id << 1 | flow_bit
// flow_bit: Forward=0, Reverse=1

// Max key: 55<<7 | 55<<1 | 1 = 7151
// Array size: [8192] (next power of 2)
// Memory: 8192 × 2 bytes = 16 KB — fits in L1 cache

type adjacencyMap [8192]uint16

// Edge flags in each uint16 slot:
//   bit 0 — flagPresent      (transition exists)
//   bit 1 — flagOverride     (FORCE_* edge)
//   bit 2 — flagRequiresOpID (needs operator_id)
//   bit 3 — flagInterSystem  (crosses ownership boundary)

// Zero value = no transition. Non-zero = valid.
// Single read: adj[packKey(from, to, flow)]
Why a fixed array, not a hash map?
A Go map costs ~50–100ns per lookup (hashing + bucket scan + pointer chasing). A fixed array costs one memory load instruction. At 50 lakh events/day, the difference is material.
Why 16 KB matters
L1 cache on any modern server CPU is 32–64 KB per core. The 16 KB array fits entirely in L1. After the first ~10 calls (warmup), every lookup is served from L1 — effectively a register-speed operation.
Zero-allocation guarantee
The success path returns a struct by value. AnomalyReason strings on the failure path are compile-time constants (data segment). No heap escapes. Enforced by testing.AllocsPerRun in CI.

Why offline is different
The delivery agent may be in a basement with no signal. Blocking status updates would freeze the entire last-mile operation. The app accepts the validated transition locally and syncs when online.
What can go wrong
The agent's app holds a stale copy of the shipment's current status. If another system moved the shipment while offline, the app's assumed prior state is wrong. This is why the tracking anomaly detector exists — it cross-checks on sync.
Rule engine still runs
Even offline, the embedded engine validates the transition locally. The agent cannot emit a structurally invalid transition — only potentially a state-inconsistent one, which the TAD catches on sync.

📄 Stage 1 — Spec Validation
What it checks
A JSON Schema validator runs against rules.yaml before any Go code is compiled. It catches: duplicate status IDs, unknown owned_by values, transitions referencing undefined status codes, wildcard targets marked as terminal, and missing required fields.
Why this runs first
If the spec is structurally invalid, every subsequent stage will also fail — but with cryptic errors. The schema validator surfaces the exact problem (line, field, value) in 2 seconds, saving engineers from debugging a Go panic.
# spec-repo CI — runs on every PR
validate-spec:
  run: ajv validate -s spec/schema.json -d spec/rules.yaml
  # exits non-zero on any violation
  # output: line number + field + error message
🏗 Stage 2 — Build Gate
Invalid spec = startup panic (before first request)
NewEngine() calls panic() if spec/rules.yaml is missing or fails to parse. This means a bad spec is caught when the process starts — before it handles any traffic. The test suite calls NewEngine() at package init, so it fails on the first go test run.
Typos are compile errors
Typed status constants (status.HubOutscan, status.Delivered) mean that source systems referencing a renamed or deleted status will fail to compile. No stringly-typed status codes escape into the codebase.
test:
  run: go test ./...
  # var engine = NewEngine() at package level
  # → panics immediately if spec/rules.yaml is invalid
  # → caught before any test function runs

⚡ Stage 5 — Zero-Alloc Gate
The most important CI gate
Any change that introduces a heap allocation on the Validate() hot path will silently degrade throughput at 50L events/day. This test makes that regression impossible to ship.
func TestValidate_NoAlloc_SuccessPath(t *testing.T) {
  e   := statusrules.NewEngine()
  ctx := statusrules.TransitionContext{
    SourceSystem: statusrules.SystemHub,
  }
  allocs := testing.AllocsPerRun(1000, func() {
    e.Validate(status.HubInScanned, status.HubOutscan,
      statusrules.FlowForward, ctx)
  })
  if allocs > 0 {
    t.Fatalf("hot path allocates: %.0f allocs/op", allocs)
  }
}
Runs on three paths
Success path · Failure path · FORCE_* override path. All three must show 0 allocs/op. A vtable dispatch, string format, or slice append anywhere on these paths will fail the gate.

⏪ Rollback Procedure
Library rollback (minor/patch)
Pin the previous library version in go.mod (or pom.xml / package.json), merge the revert PR, rolling restart. Rolling back the commit reverts both the library and spec/rules.yaml together — no config store to revert separately. Typical rollback time: same as a normal deploy.
Major version rollback (breaking change)
All four source systems must roll back in the same window. Because the old spec rejects transitions the new spec added, there is no safe partial rollback. Coordinate the revert deploy across all systems before any traffic is shifted back.
Rollback does NOT require a separate spec revert
All four language packages live in the same monorepo as spec/rules.yaml. Rolling back to a prior commit reverts the spec and all language implementations together atomically. No separate spec repository to coordinate.

Why tolerate 2 minor versions of skew
Source systems have their own release cycles. Hub Ops may deploy weekly; the Delivery mobile app may deploy monthly. A hard rejection on stale spec versions would cause operational outages during normal deployment windows. The 2-minor-version window gives teams enough runway to upgrade without urgency.
What the on-call alert triggers
When a source system is >2 minor versions behind, the tracking ingestion layer fires a low-severity alert to that system's on-call channel. It is not a page — it is a ticket to upgrade. Repeated alerts on the same system escalate to the Platform team.

The golden rule: spec changes are schema migrations
Every status transition rule is load-bearing infrastructure. Removing or changing a transition is as consequential as dropping a database column. The governance process exists to ensure that operational teams, not just engineers, sign off on changes that affect real shipment flows.
